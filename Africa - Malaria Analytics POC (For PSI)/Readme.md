This project is a PSI-aligned malaria program performance pipeline MVP built to showcase end-to-end cloud data engineering and analytics for public health decision-making across 18 Sub-Saharan African countries. It focuses on tracking malaria burden and intervention coverage, then converting these data into portfolio-ready KPIs and priority tiers that can direct PSI’s resources toward the highest-need geographies.

The solution runs on Microsoft Fabric using a Medallion (Bronze–Silver–Gold) delta lakehouse architecture, with managed Delta tables in the Tables/ area so everything is queryable via a SQL endpoint and exposed cleanly to Power BI through DirectLake. All core data flows through API-based ingestion instead of manual file drops, so the pipeline is realistic and production-leaning rather than a CSV toy example.

Upstream, the pipeline pulls from four main sources: WHO GHO for malaria incidence, deaths, and intervention coverage; WHO Africa Health Observatory for health system context; World Bank Open Data for macro indicators such as health expenditure and GDP; and a small synthetic PSI-style program dataset to approximate beneficiaries reached, spend, and basic efficiency metrics. These are ingested into Bronze Delta tables with explicit schemas, run IDs, and minimal assumptions, preserving a faithful record of the raw source.

In the Silver layer, the project standardizes country codes and names, constrains year ranges, cleans and joins indicators, and enforces data quality rules such as valid ranges, non-null joining keys, and removal of obvious duplicates. This layer produces clean, analysis-ready health and malaria fact tables along with country and indicator dimensions that can be reused by multiple downstream products.

The Gold layer defines a star schema centered on gold_fact_country_kpi and a slowly changing dim_country, plus a DQ audit table that tracks data quality at each run. Here the project computes composite scores using min–max normalization (0–100 scale) to put epidemiological, economic, and programmatic metrics onto a comparable footing.

Four main KPIs drive the decision logic: a MalariaRiskScore combining normalized malaria incidence and deaths; a HealthSystemScore derived from health expenditure and GDP per capita; a PSIProgramScore summarizing program efficiency and coverage quality; and an OverallPSIScore that weights these components so high disease burden penalizes performance and strong systems/programs offset it. The score distribution is calibrated over the actual country-year portfolio to yield intuitive priority tiers: Priority 1 (Immediate Action), Priority 2 (Accelerate), Priority 3 (Monitor), and On Track.

The repository is structured for hiring managers and technical reviewers, with a clear separation of concerns. It includes notebooks for Bronze ingestion, Silver transformation, and Gold analytics; shared utilities for API calls, schema management, and data quality checks; and tests that validate key transformation logic in the Silver layer.

Documentation is treated as a first-class asset. A detailed README explains the business problem, high-level architecture, and how to set up and run the project; an SOP documents operational expectations (scheduling, failure handling, and governance); and a data dictionary spells out business and technical definitions for all tables and fields.

There is also guidance for orchestration and reporting. An Azure Data Factory-focused document explains how to schedule and monitor the pipeline, while Power BI documentation walks through the semantic model, DAX measures, and dashboard layout used to tell the malaria program story.

On the analytics side, the project connects Gold tables to Power BI via DirectLake and builds a multi-page dashboard aligned with PSI’s brand colors (#73a534 and #f7941e). The dashboard includes an executive overview (headline scores, risk trend, highest-risk countries), an intervention coverage view (ITN/ACT/IRS/IPT coverage versus WHO targets), and a PSI program performance page (beneficiaries, spend, efficiency, and data quality status).

Throughout, the project is explicit that PSI program metrics are synthetic and for demonstration only, with transparent notes so a non-technical reviewer understands what is real and what is illustrative. The scoring model itself went through several iterations to fix unit mismatches and unrealistic ranges, and the final version is backed by numeric checks on score distributions, row counts, and null patterns so the portfolio tiers look and behave like something PSI could actually use.
